# Source Metrics Method for `.mts` Code

- Status: active operator note
- Scope: documentation-only; no runtime, build, or verification-script behavior changes

## Purpose

The repository's TypeScript source of truth lives in `.mts` files under `src/`, while committed `.mjs` files under `bin/`, `lib/`, `hooks/`, and `test/` are generated by `tsc`. Generic codebase-size reports can therefore mislead future agents if they only quote `pygount`'s detected-code totals.

Use this note when reporting repository size, source-vs-generated composition, or trends in AI-improvement logs.

## Recommended reporting contract

1. Use `pygount` for broad language/distribution scans because it still captures generated JavaScript, Markdown, JSON, YAML, Bash, HTML, binaries, duplicates, and overall file counts.
2. Do **not** describe the `pygount` `Code` total as the repository's TypeScript source size. In this repo, the main source language appears under `__unknown__` when scanned as `.mts`.
3. When source-size accuracy matters, pair the `pygount` summary with the supplemental Node `.mts` scan below.
4. Label the supplemental count as **total source lines** or **non-empty source lines**, not pygount-style executable code lines. The Node scan is intentionally simple and does not strip comments.
5. Re-run both commands after documentation-heavy edits if the report quotes Markdown/comment totals, because AI-improvement logs and plan notes change the Markdown comment count.

## Commands

From `C:/Ai/continuous-improvement`:

```bash
uvx --from pygount pygount --format=summary \
  --folders-to-skip='.git,node_modules,venv,.venv,__pycache__,.cache,dist,build,.next,.tox,.eggs,*.egg-info,third-party' \
  .
```

Proof that `.mts` is classified as unknown by `pygount` in this environment:

```bash
uvx --from pygount pygount --suffix=mts --format=summary src
```

Supplemental `.mts` source scan:

```bash
node --input-type=module -e '
import { readdirSync, statSync, readFileSync } from "node:fs";
import { join, relative, sep } from "node:path";
const byDir = new Map();
let files = 0;
let lines = 0;
let nonEmpty = 0;
function walk(dir) {
  for (const name of readdirSync(dir)) {
    const path = join(dir, name);
    const stat = statSync(path);
    if (stat.isDirectory()) {
      walk(path);
      continue;
    }
    if (!path.endsWith(".mts")) continue;
    files += 1;
    const text = readFileSync(path, "utf8");
    const rows = text.length === 0 ? [] : text.split(/\r?\n/);
    const nonEmptyRows = rows.filter((line) => line.trim().length > 0).length;
    lines += rows.length;
    nonEmpty += nonEmptyRows;
    const parts = relative(process.cwd(), path).split(sep);
    const key = parts.slice(0, 2).join("/");
    const prev = byDir.get(key) || { files: 0, lines: 0, nonEmpty: 0 };
    prev.files += 1;
    prev.lines += rows.length;
    prev.nonEmpty += nonEmptyRows;
    byDir.set(key, prev);
  }
}
walk("src");
console.log(JSON.stringify({
  mtsFiles: files,
  mtsLines: lines,
  mtsNonEmptyLines: nonEmpty,
  byDir: Object.fromEntries([...byDir].sort()),
}, null, 2));
'
```

## Non-goals

- Do not add a package dependency just for metrics.
- Do not add an npm script until maintainers repeatedly need this outside AI-improvement logs.
- Do not modify generated `.mjs` files or plugin artifacts for metrics documentation.
- Do not replace `pygount`; augment it where `.mts` source counts matter.

## Future trigger for a code/script change

If maintainers start requesting this count in releases, CI reports, or contributor docs, promote the supplemental scan into a small no-dependency script and test it. Until then, this operator note is the smallest useful fix.
