# 2026-06-27 — `verify:skill-count-prose` invariant + roast count sync

## Goal

Make the hand-maintained skill-count prose impossible to drift from the actual bundle. PR #255 (`/roast`, the 26th skill) bumped only the *gated* "N bundled skills" manifest surfaces and left the human-facing prose at "25 skills / 14 tier-2." This plan ships the doc sync **and** a new lint so the same lag can never pass CI green again.

## Root cause

`verify:skill-count` ([bin/check-skill-count.mjs](../../bin/check-skill-count.mjs)) only asserts the literal phrase `N bundled skills` in 5 manifest files + the landing stat block. The human docs (`README.md`, `docs/skills.md`, `docs/skill-use-cases.md`, `CONTRIBUTING.md`, `docs/using-this-plugin.md`) use different phrasings — "26 skills", "26-skill catalog", the `1 core + 1 featured + 6 tier-1 + 15 tier-2 + 3 always-bundled` breakdown, and the "Adding a 27th skill" ordinal — none of which any invariant checks. So they silently rot. `reports/daily-improvement.md` logs three prior rounds of exactly this drift.

## Source of truth

`skills/*.md` + the root `SKILL.md`, via the existing primitives:
- `discoverSkillSources(repoRoot)` — exported from [src/bin/check-skill-tiers.mts](../../src/bin/check-skill-tiers.mts); returns the core skill + every `skills/*.md` (README excluded).
- `parseSkillFrontmatter` + `normalizeTier` — from [src/lib/skill-tiers.mts](../../src/lib/skill-tiers.mts).

Derived counts today: `core=1, featured=1, tier-1=6, tier-2=15, companion=3 → total=26`, `next=27`. **Nothing is hardcoded** — the lint recomputes from frontmatter on every run.

## Design — one lint covers both WILD items

WILD item 1 (gate the "N skills" phrasing) and WILD item 2 (derive the breakdown from frontmatter and fail any disagreeing doc) collapse into a **single** lint: `bin/check-skill-count-prose.mjs` (source `src/bin/check-skill-count-prose.mts`).

Exported, testable functions:
- `deriveCounts(repoRoot)` → `{ core, featured, tier1, tier2, companion, unknown, total, next }`.
- `ordinal(n)` → e.g. `27 → "27th"` (handles 11–13 → "th").
- `expectedFor(check, counts)` → builds the expected substring for a check kind.
- `checkProseSurfaces(repoRoot, counts?, surfaces?)` → `Array<{ file, check, expected, reason }>` (`reason`: `stale` | `target-missing`).
- `main()` — print `OK`/`FAIL`, exit 0/1; guarded so importing the module never runs it.

Check kinds → expected substring:
- `breakdown` → `${core} core + ${featured} featured + ${tier1} tier-1 + ${tier2} tier-2 + ${companion} always-bundled`
- `total-skills` → `${total} skills`
- `total-catalog` → `${total}-skill catalog`
- `next-ordinal` → `${ordinal(next)} skill`

Surfaces manifest (the spec):

| File | Checks |
|---|---|
| `README.md` | breakdown, total-skills, next-ordinal |
| `docs/skills.md` | breakdown, total-skills, total-catalog, next-ordinal |
| `docs/skill-use-cases.md` | breakdown, total-skills |
| `CONTRIBUTING.md` | total-skills |
| `docs/using-this-plugin.md` | total-catalog |

## Tests — `src/test/check-skill-count-prose.test.mts`

Boundary-first (CLAUDE.md 2026-06-03: green gates do not prove boundary safety):
1. Happy path — real repo derives `total=26` and `checkProseSurfaces` returns zero failures.
2. `deriveCounts` on a temp fixture with known tiers returns the exact breakdown.
3. Stale total — a planted "25 skills" is flagged with file + expected.
4. Stale breakdown — a planted "14 tier-2" is flagged.

6. Missing surface file → `target-missing`, not a crash.
7. `ordinal` — 1→1st, 2→2nd, 3→3rd, 11→11th, 21→21st, 27→27th.
